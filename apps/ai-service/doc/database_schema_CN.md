# DRR 框架：数据库设计与存储精华

## 1. 存储哲学
DRR (Tree-Graph Dual Representation Reasoning) 采用的是一种**“三位一体”的思想**：关系型存储提供工程数据的严谨性，向量型存储提供语义搜索的灵活性，图谱存储提供全局知识的联通性。系统底层依托 **PostgreSQL**，并集成 **pgvector** 扩展，实现了这三种模态的无缝融合。

---

## 2. 数据库中的技术精华 (The "Gems")

### 2.1 递归 CTE 机制溯源 (Recursive CTEs)
这是本系统在数据库层面的核心杀手锏。生物机制的演化往往是逐级嵌套的（例如：动物器官 -> 组织微观结构 -> 物理化学特性）。
- **创新点**：传统 RAG 需要在应用层进行多次数据库往返（N+1 查询）来构建机制层级，这在面对海量知识图谱时会造成严重的 I/O 阻塞。
- **技术解决**：我们利用 Postgres 的递归公用表表达式（CTE），在一行 SQL 中完成对机制树的完整溯源，极大地提升了提取效率。

### 2.2 混合向量索引 (Hybrid Vector Indexing)
系统针对高维语义搜索进行了极致优化。
- **技术亮点**：采用 **HNSW (Hierarchical Navigable Small World)** 算法。不同于简单的 IVFFlat，HNSW 在极高精度下依然能保持毫秒级的检索速度。
- **动态分片**：根据机制树的 `level` 字段对索引进行动态加权，确保浅层的功能描述（如“轻质”）与深层的结构细节（如“纳米纤维”）具备不同的检索优先级。

### 2.3 Leiden 拓扑社区索引
在 GraphRAG 层面，数据库不只是存储数据，更是在“理解数据”。
-   **技术细节**：利用 **Leiden 社区发现算法** 对所有生物学实体进行预聚类。
-   **优势**：在执行“全局搜索”时，系统不再逐个搜索节点，而是直接检索相关的**语义社区 (Semantic Communities)**。例如，如果查询涉及“流体力学”，系统会直接定位到“水生节肢动物”和“鸟类飞翔动力学”这两个高频社区。

## 3. 技术挑战与优化方案

### 3.1 跨语言 Embedding 映射
-   **挑战**：科研文献往往以英文为主，而用户查询可能包含中文。
-   **方案**：我们在数据库层实现了多语言向量中转。通过对 Embedding 向量空间进行归一化处理，实现了“中英语义同源搜索”。

### 3.2 自动清理与索引重建 (Auto-Vacuuming for AI)
-   **优化**：针对频繁生成的机制临时节点，系统配置了精细化的 Vacuum 策略，确保 pgvector 索引在不停机的情况下保持最佳检索拓幅。

---

## 4. 结论
DRR 框架的数据库设计不仅满足了存储需求，更通过深度优化 SQL 层面的层级提取，将数据库转化为一个具备“逻辑内省能力”的知识引擎。它为大规模自动化的学术发现提供了最稳固的底座。
